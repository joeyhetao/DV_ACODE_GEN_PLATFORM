根据代码实际状态，将架构变更同步到 ARCHITECTURE.md，将需求变更同步到 PRD.md。

用户可通过 `$ARGUMENTS` 传入变更摘要，例如：
`/update-specs 新增了 WebSocket 实时推送，移除了轮询机制`

若未传入参数，则从 git log 和代码状态自动推断变更。

## 执行步骤

### 第一步：采集变更信息

**运行以下命令**，收集代码实际状态：

```bash
git log --oneline -30
git show --stat HEAD
git diff HEAD~1 HEAD --name-only 2>/dev/null || git diff --name-only --cached
```

同时读取关键实现文件，判断架构事实：
- `backend/app/api/v1/router.py`（当前注册的所有路由）
- `backend/app/services/core/pipeline.py`（生成流水线步骤）
- `backend/app/core/config.py`（环境变量和配置项）
- `backend/app/models/` 下所有模型文件（数据库表结构）
- `docker-compose.yml`（服务组成）
- `embedding_service/app/main.py`（Embedding Service 接口）

**如果用户提供了 `$ARGUMENTS`**，将其作为变更摘要的起点，结合代码验证。

---

### 第二步：判断哪份文档需要更新

对每处发现的差异，按以下规则分类：

| 差异类型 | 更新目标 |
|---------|---------|
| 新增/修改 API 端点、服务接口、数据库表/字段 | ARCHITECTURE.md |
| 目录结构变更、新增服务容器、环境变量变更 | ARCHITECTURE.md |
| 缓存策略、队列配置、向量检索参数变更 | ARCHITECTURE.md |
| 新增用户可见功能、权限变更、交互流程变更 | PRD.md |
| 非功能需求调整（性能指标、安全约束等） | PRD.md |
| 功能从 Out of Scope 变为已实现，或反向 | PRD.md |
| 架构实现与 PRD 功能描述均需更新 | 两份都更新 |

若某项变更与两份文档均无关（纯 bug fix、代码重构、前端样式），则跳过文档更新，在最后说明原因。

---

### 第三步：更新 ARCHITECTURE.md

**注意**：ARCHITECTURE.md 超过 35k tokens，必须用 offset+limit 分段读取，不要一次性读取整个文件。

按以下顺序操作：

#### 3a. 读取文件头部（版本信息，约前 20 行）
确认当前版本号（格式 `v2.X`）和最近变更条目。

#### 3b. 按需读取受影响的章节
根据第二步的分类，只读取需要修改的章节：
- §1 架构总览：修改架构图时读取（offset 约 20-150 行）
- §3 各层设计（含 §3.x）：修改服务实现时读取
- §4 数据库设计：表结构变更时读取
- §5 API端点：路由变更时读取
- §7 目录结构：文件结构变更时读取
- §8 部署配置：Docker/环境变量变更时读取

#### 3c. 执行修改

对每个需要更新的章节，精确 Edit，只改有差异的部分：

**常见修改场景**：

1. **新增 API 端点**：在 §5.1 端点列表中追加对应行
2. **新增数据库字段**：在 §4.1 对应表的字段列表中追加
3. **新增环境变量**：在 §8.4 环境变量表中追加，含注释说明
4. **目录结构变更**：更新 §7 中的目录树，与实际 backend/ 结构对齐
5. **服务流程变更**：更新对应 §3.x 中的流程描述或流程图
6. **新增容器服务**：更新 §8.1 Docker Compose 服务说明

#### 3d. 更新版本头

在文件头部的变更日志追加一条：
```
- vX.Y → vX.Z：<一句话描述本次架构变更>（§N.M）
```
版本号：当前版本 +0.1（如 v2.10 → v2.11）。
同时将 `**版本**` 和 `**日期**` 字段更新为新版本和今天日期。

### 第四步：更新 PRD.md

PRD.md 约 720 行，可完整读取。

#### 4a. 读取完整 PRD

#### 4b. 按需修改章节

**常见修改场景**：

1. **新增功能需求**：在对应 §3.x 章节中追加功能描述，或新建 §3.x 子节
2. **权限矩阵变更**：更新 §3.6 权限矩阵表格

在文件头部的变更条目追加：
```
- vX.Y → vX.Z：<一句话描述本次需求变更>（§N.M）
```
版本号：当前版本 +0.1（如 v2.7 → v2.8）。
同时更新 `**版本**` 和 `**日期**`。

### 第五步：提交

两份文档修改完成后：

```bash

```

若只有一份文档有变更，只 add 那一份。

## 修改原则

- **只改有差异的部分**：风格、语言（中文）、章节结构保持不变，不重写现有正确内容
- **不推测**：若某段代码意图不明，读取源文件确认后再写，不要凭假设更新文档
- **保持版本号单调递增**：ARCHITECTURE 和 PRD 各自独立维护版本号，互不影响
- **架构图用 ASCII**：若需修改 §1 中的架构图，保持原有 ASCII 框图风格
- **不改 PRD 的用户故事**：PRD §1-§3 描述的是"用户视角的需求"，不要把代码实现细节写进去
- **不改 ARCHITECTURE 的 PRD 内容**：ARCHITECTURE 描述"如何实现"，不重复 PRD 的"需要什么"
